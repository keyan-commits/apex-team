# test-coverage-audit

The invariant this skill enforces:

> **QA MUST map every `requirements/features/FEAT-NNNN-*.md` (and its child US acceptance criteria) to the tests that assert it, audit each AC for gaps, and resolve or log every gap before issuing a PASS verdict.**

This applies to ANY project the eight role subagents drive — apex-team, downstream host workspaces (LFM, etc.), third-party repos. The skill is user-scoped (`~/.claude/skills/`) and so are the agents; the discipline travels.

## Why this exists

Coverage rules existed before (comprehensive-testing for positive/negative/edge + sample iteration; qa-artifact-discipline for render-and-look + real-artifact + WCAG + S1–S9). What was missing: a **systematic audit pass** that walks every FEAT, lists every AC, and asks "is there a test that proves this AC holds — and does that test satisfy the existing rules?" Audits surface unwritten tests, happy-path-only assertions, and ACs that drifted past testing entirely.

The LFM order-sheet engagement (2026-06-02) had nine distinct user-visible bugs slip past PASS — every one was an AC that lacked the appropriate test class. Without a systematic gap audit, the implementer ships PASS on what they happened to test; the operator finds what wasn't.

## Procedure

### a — DISCOVER runners + emit `scripts/test-all.sh`

Scan the workspace for build files and derive runner commands. Project-AGNOSTIC — never hardcode one project's commands; derive from what's present:

| Build file found | Runner command |
|---|---|
| `pom.xml` | `mvn test` (or `mvn -q test` for quieter output) |
| `build.gradle` / `build.gradle.kts` | `./gradlew test` |
| `package.json` with `scripts.test` | `<pm> test` where `<pm>` ∈ pnpm/yarn/npm (detect via lockfile) |
| `package.json` with `vitest`/`jest` dep but no `scripts.test` | `<pm> vitest run` or `<pm> jest` |
| `playwright.config.{ts,js,mjs,cjs}` | `npx playwright test` |
| `pytest.ini` / `pyproject.toml` (`[tool.pytest]`) | `pytest` |
| `go.mod` | `go test ./...` |
| `Cargo.toml` | `cargo test` |
| `mix.exs` | `mix test` |

For each detected runner, record `(label, command, cwd)`. Nested sub-projects each get their own entry (e.g. a monorepo with `web/package.json` + `api/pom.xml` produces two entries).

Generate or refresh `scripts/test-all.sh` at the workspace root:

```sh
#!/usr/bin/env bash
# Auto-generated by test-coverage-audit skill. DO NOT EDIT BY HAND.
# Re-run the skill on its FEAT to regenerate after adding new sub-projects.
set -e
echo "── apex-team test-all ────────────────────────────────────────"
echo "Workspace: $(pwd)"
echo

# <runner-entry-1>
echo "▶ <label-1> (<cwd-1>)"
( cd "<cwd-1>" && <command-1> )
echo

# <runner-entry-2>
echo "▶ <label-2> (<cwd-2>)"
( cd "<cwd-2>" && <command-2> )
echo

echo "── all runners green ─────────────────────────────────────────"
```

`chmod +x scripts/test-all.sh` so the operator can run the whole suite with one command.

### b — MAP each FEAT → tests

For each `requirements/features/FEAT-NNNN-*.md`:

1. Parse its frontmatter to get FEAT id + slug.
2. Find child user stories via `parent_feat:` lookups across `requirements/user-stories/`.
3. For each US, parse its `## Acceptance criteria` section to enumerate ACs (numbered or bulleted).
4. Locate the test file(s) at `tests/qa/features/FEAT-NNNN-<slug>/TEST-NNNN-*.test.{ts,js}` (or per-runner equivalents: `*Test.java`, `*_test.go`, `test_*.py`).
5. Within each test file, identify which AC numbers each `it()` / `test()` / `@Test` block asserts (by name / comment / explicit AC tag in the description).
6. Build the matrix: `(FEAT, US, AC) → [test_function_names...]`.

### c — AUDIT gaps with the rubric

For each AC in the matrix, check:

1. **At least one positive test asserts the happy path.** (comprehensive-testing class 1)
2. **At least one negative test asserts the rejection / error path.** (class 2)
3. **At least one edge-case test on every input axis the AC exposes.** (class 3)
4. **Full sample-set iteration** for any AC that depends on parsing/processing/rendering files from a known-input directory (class 4 — iterate every file under `requirements/samples/**`, not a "representative").
5. **S1–S9 gates** for any AC that produces a visual or operator artifact:
   - S1 render-and-look | S2 real-artifact end-to-end | S3 realistic data | S4 positional/semantic | S5 WCAG contrast | S6 reference diff | S7 validated≠deployed | S8 question business intent | S9 human-meaningful assertion
6. **AC traceability** — the test's description / name / comment cites the AC number, so the matrix can be audited mechanically.

Flag every AC with:
- ❌ **No test** — the AC has zero asserting tests.
- ⚠ **Happy-path only** — at least one positive but no negative / edge / iteration.
- ⚠ **Iteration gap** — sample directory exists but the test doesn't loop over every file.
- ⚠ **Artifact gate skipped** — the AC produces a visual/operator artifact but S1+S2 didn't apply (no rendered inspection / no real-path verification).
- ⚠ **AC drift** — AC exists in US but no test cites it.

### d — REPORT

**Per-FEAT report:** write `requirements/features/FEAT-NNNN-<slug>/TEST-PLAN.md` (or update if extant) with a `## Coverage & Gaps` section listing every AC + its current coverage + every gap. Each gap is an actionable line ("AC3 needs a negative test for malformed-frontmatter input — see `tests/qa/features/FEAT-NNNN-<slug>/TEST-NNNN-*.test.ts:N`").

If `TEST-PLAN.md` doesn't exist for the FEAT, create it with the standard sections (Scope, AC matrix, Coverage & Gaps, Open questions).

**Workspace-level summary:** write or refresh `tests/qa/COVERAGE-AUDIT.md` at the workspace root with:
- Run timestamp + tool version.
- Per-FEAT counts: total ACs, ACs covered, ACs with gaps.
- Top-N actionable items: the most load-bearing gaps to close next (sorted by FEAT priority + AC criticality).
- Anti-gap counter-claims: any AC where coverage exceeds the rubric (e.g. cross-fixture iteration that goes beyond the minimum) — credit the coverage explicitly so reviewers don't double-count.

The two reports together let QA, BA, and any reviewer see "what tests exist, what they prove, and what's still missing" in one read.

## When to invoke

**Auto-fire on QA work:** the `.claude/agents/qa.md` body carries an auto-fire hook that says: *"After authoring or revising tests for any FEAT, invoke the `test-coverage-audit` skill and resolve or log every gap in COVERAGE-AUDIT.md before issuing a PASS verdict."*

**Manual invocation:** QA may also run the audit at any time to refresh the reports. Re-running is idempotent — gaps that close drop off the report; gaps that persist remain flagged.

**Cadence cue:** run the audit at every wave-close before QA emits its PASS verdict. The audit is the source of truth for whether the wave's tests cover the wave's ACs.

## Anti-patterns this skill prevents

1. **Single-runner blindness** — assuming the whole workspace is pnpm/vitest when there's a Java sub-project sitting next door uncovered. The discover-and-emit pass surfaces every runner in the workspace.
2. **AC drift** — an AC documented in the US but never traced to a test. The matrix step catches this mechanically.
3. **Happy-path PASS** — the implementer wrote one test that passes; the audit flags the missing negative/edge/iteration classes.
4. **Mocked-artifact verification** — for visual/operator deliverables, the audit checks whether S1+S2 applied. A "PASS" without render-and-look is an audit fail.
5. **Silent gap accumulation** — multiple waves ship without an audit; gaps accumulate invisibly; an incident reveals the coverage debt. The auto-fire hook prevents accumulation.
